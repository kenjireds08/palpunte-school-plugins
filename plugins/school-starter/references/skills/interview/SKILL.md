# Interview Skill

共通理解に達するまでインタビューし、要件定義書を生成するスキル。
**「要件定義が9割」の思想**で、曖昧さを残さず、用語の揺れを潰し、技術者向けとクライアント向けの2種類の要件定義書を生成する。

出力は `/feature-dev` や `/deep-plan` にそのまま渡せる形式。

## --full モードの4フェーズ構成

| フェーズ | 内容 |
|---------|------|
| 1. コードベース探索 | 既存プロジェクトのパターン・DB・CLAUDE.md を把握 |
| 2. 徹底インタビュー | 9観点 + Grill Me 風深掘りで曖昧さを残さず潰す |
| 2.5 用語集確定 | Ubiquitous Language で揺れを潰す（予約 vs 申し込み等） |
| 3. 技術者向け仕様書出力 | `docs/NNN_requirements.md` 生成・ブラッシュアップ |
| 4. クライアント向け仕様書生成（要求時） | 粒度選択(A/B/C/カスタム) → `_for_client.md` 生成 |

## モード選択

引数でモードを切り替える:

- `--light`: **モックアップ・見積もり・商談用** の軽量モード（5問で完結）
- `--full`（デフォルト）: **本実装用** の詳細モード（既存5観点 + コスト・法令・LLMリスク・性能）

### 引数なしで呼ばれた場合の判断
- `docs/spec-light-*.md` が既にある → **--full モード**で軽量仕様を補完して詳細化
- 新規プロジェクトかつモックアップ前 → どちらを使うかユーザーに確認
- それ以外 → **--full モード**で実行

## フェーズ1: コードベース探索（既存プロジェクトの場合・両モード共通）

インタビューを始める**前に**、まずコードベースを探索する。

- プロジェクト構造、既存パターン、技術スタック、DB構成を把握
- CLAUDE.md、000_PROJECT_STATUS.md を確認
- **コードを読めば答えが出る質問は、ユーザーに聞かずに自分で調べる**

探索で得た情報は、インタビューの質問に反映する。
「このプロジェクトではSupabaseを使っていますが、この機能もSupabaseで実装しますか？」のように具体的に聞く。

## ▼ --light モード（モックアップ・商談用）

### 目的
モックアップ作成・見積もり作成・商談に必要な**最低限の情報**を集める。
DB詳細設計・セキュリティ実装詳細・パフォーマンス最適化は聞かない（受注後に `--full` で詰める）。

### 質問構成（5つの AskUserQuestion で完結）

#### Q1: 用途とクライアント文脈
- **これは「個人開発・趣味・プロトタイプ」か「他者が使うproduction運用」か?**（Hobby or work? = 最重要前提）
- クライアント業種・サービス内容
- クライアント規模（個人事業主 / 数店舗 / 中小企業 / 大手）
- 現状の運用方法（電話、紙、Excel、既存ツール）
- 月間想定件数・利用規模の概算

#### Q2: エンドユーザー
- 利用者の年齢層・技術リテラシー
- 利用デバイス（スマホ中心 / PC中心 / 混在）
- 利用シーン（事前予約 / 当日利用 / リマインド前提など）

#### Q3: 機能スコープ（MVP）
- 顧客側で必須機能（リスト化）
- 管理画面の機能（リスト化）
- **今回やらないこと**（スコープを明確に切る。例: 決済、会員機能、レビュー）

#### Q4: モックアップ範囲
- 何画面見せたい? （3〜5画面が現実的）
- 主要画面リスト（例: TOP / 予約フォーム / マイページ / 管理ダッシュボード）
- デザインの方向性・参考サイト

#### Q5: 制約・前提
- おおまかな予算感
- おおまかな納期感
- 技術的な希望・制約（既存システム連携、特定サービス指定など）

## ▼ --full モード（本実装用）

### 目的
本実装前に、要件・データ・セキュリティ・コスト・法令を**全て固める**。

### 既に `docs/spec-light-*.md` がある場合
そのファイルを読み込み、**共通項目はスキップして不足分のみヒアリング**する。
「Q1の用途・クライアント情報は spec-light で確認済みです。次に詳細を聞きます」のように案内する。

### フェーズ2: 徹底インタビュー

AskUserQuestionツールを使い、設計ツリーの各分岐を1つずつ歩く。

#### 質問の進め方（要件定義が9割。ここで手を抜かない）

1. **まず全体像を確認する** — 何を作りたいか、なぜ作るか、誰が使うか（spec-lightがあればスキップ）
2. **設計ツリーの分岐を歩く** — 各判断ポイントで選択肢を提示し、依存関係を1つずつ解決する
3. **曖昧な回答は放置しない（最重要）** — ユーザーの回答が抽象的・曖昧な場合、次の観点に進まず**その場でデシジョンツリーの枝を潰す**
   - 例1: 「ユーザー登録はメール認証で」
     → 「パスワードリセット方式は？」
     → 「既存アカウントの重複検出は？」
     → 「退会時のデータ処理は？（物理削除 / 論理削除 / 匿名化）」
     → 「未確認メールの扱いは？」
   - 例2: 「予約をキャンセルできるように」
     → 「キャンセル期限は何時間前まで？」
     → 「キャンセル料は発生する？」
     → 「キャンセル後の枠は別の顧客が取れる？」
     → 「既に支払い済みの場合の返金フローは？」
   - **抽象的な語（「いい感じに」「普通に」「ちゃんと」等）は必ず具体化を要求**する
4. **仮説を立ててぶつける** — ユーザーが判断を迷ったら、こちらから選択肢を提示
   - 「通常はX（大多数のサービスで採用）、ただしこのケースもありえます: Y。どちらが近いですか？」
   - 迷いを放置すると後工程で推測実装されてズレる
5. **共通理解に達するまで繰り返す** — 実装に必要な判断がすべて済むまで止めない
   - 目安: **1つの観点で最低3-5往復**のやりとりが発生することも普通（むしろそれが健全）
   - 講師（菊池）の流儀: **要件定義に2-3日かけてでも妥協しない**

#### 質問の観点（9項目: 既存5 + 新規4）

##### 既存の5観点
- ユーザー体験（誰が、いつ、どう使うか）
- データ設計（何を保存するか、関連は、制約は）
- 技術的判断（認証、API設計、外部サービス連携）
- エッジケース（エラー時、データ不整合、権限）
- 優先順位（MVP vs フル機能、段階的リリース）

##### 新規追加の4観点（general-prerequisitesより）
- **コスト**: 外部API課金（OpenAI、Stripe、Supabase等）、polling loop、想定スケール時の月額試算、ハード上限の有無、kill switchの設計
- **法令・コンプライアンス**: 個人情報の取り扱い、GDPR/APPI、対象業種の規制（医療・金融・教育）、依存ライブラリのライセンス
- **LLMリスク（該当時のみ）**: データ経路（どのプロバイダ・どのリージョン）、Training opt-out状況、Prompt injection対策、LLMに与えるツール権限の範囲
- **性能の現実**: 1年後・3年後のデータ量見積もり、N+1パターンの予防、index設計、想定QPS

#### ルール
- 「当たり前すぎる質問」はしない（コードを見ればわかることは聞かない）
- 1回のAskUserQuestionで関連する質問を2〜4個まとめて聞く（1問ずつ聞かない）
- ユーザーが「完了」「もう大丈夫」と言うか、設計判断がすべて解決するまで続ける
- 判断に迷う箇所は選択肢を提示して選んでもらう
- LLMリスクは**LLMを使う案件のみ**問う（一般的なCRUDアプリでは不要）

### フェーズ2.5: Ubiquitous Language（用語集）の確定

インタビューがほぼ完了した段階で、**用語の揺れを潰すフェーズ**を実行する。
クライアント・開発者・AIで**同じ言葉が同じ意味**になるよう、用語集を確定する。

#### やること

1. **インタビュー中に出てきた業務用語・UI用語・データ用語を洗い出す**
   - 例: 「予約」「申し込み」「キャンセル」「編集」「確定」「仮予約」「ユーザー」「お客様」「会員」「管理者」「スタッフ」「オーナー」
2. **揺れを検出**
   - 「予約」と「申し込み」が混在していないか
   - 「ステータス」が予約ステータスと注文ステータスで二重定義されていないか
   - 「ユーザー」が顧客側と管理側の両方を指していないか
3. **揺れがあれば AskUserQuestion で統一を確認**
   - 「このプロジェクトでは『予約』と『申し込み』のどちらで統一しますか？」
   - 「『キャンセル』と『取り消し』は同じ意味ですか？別の概念ですか？」
4. **用語集を確定**
   - 各用語に「**意味**」「**使われる文脈**」「**類義語があれば使わない旨**」を明記
   - 仕様書の末尾に必ず **「用語定義（Ubiquitous Language）」セクション**として入れる

#### 例

| 用語 | 意味 | 備考 |
|------|------|------|
| 予約 | 顧客がサービスを事前に確保する行為 | 「申し込み」は使わない |
| キャンセル | 予約確定後の取り消し | 確定前の変更は「編集」と呼ぶ |
| ステータス | 予約の状態（予約済 / キャンセル / 完了） | 注文ステータスとは別概念 |
| 顧客 | サービスを利用するエンドユーザー | 「ユーザー」はシステム利用者一般を指すため混同注意 |

**このセクションは技術者向け・クライアント向けの両方の仕様書に含める**。

### フェーズ3: 技術者向け仕様書出力

インタビュー完了後、技術者向け（Claude Code が読みやすい・実装指示として使える）仕様書を出力する。

#### ファイル名規則（講師の推奨運用準拠）

- **新規プロジェクト全体の要件定義**: `docs/001_requirements.md`
- **機能追加の要件定義**: `docs/NNN_[機能名]_requirements.md`
  - NNN は既存の `docs/` 配下の番号を探索して**次の番号**を付ける
  - 例: 既存が `001_requirements.md` ~ `006_task_backlog.md` なら、次は `007_[機能名]_requirements.md`
- 既存の `spec-[機能名].md` 形式も互換で受け入れる（新規生成は番号付きを推奨）

#### テンプレート

仕様書出力には以下を含める（既存項目 + 用語集セクション追加）:

```markdown

- LLM利用: あり / なし

## 背景・目的

- 誰が使うか
- 成功の定義

|-------------|------|------|
| （例: 認証方式） | （例: Supabase Auth） | （例: 既存と統一） |
```

### フェーズ4: クライアント向け仕様書生成（要求時のみ発動）

技術者向け仕様書が完成してブラッシュアップが済んだあと、**ユーザーからの明示的な要求**で発動する。

#### 発動トリガー（自然言語）
- 「クライアント向けにまとめて」
- 「Notionに送れる形にして」
- 「Googleドキュメントに貼れる形で」
- 「読みやすい形で」
- 「顧客提出用にまとめて」
- 「クライアント用の要件定義作って」

#### フロー

**Step 1**: 対応する技術者向け仕様書を Read（例: `docs/001_requirements.md`）

**Step 2**: AskUserQuestion で**粒度を選択**してもらう

```
クライアント向け仕様書の粒度を選んでください:

A. 営業・意思決定者向け（最もやさしい・見開き1枚レベル）
   - 技術用語ゼロ
   - 「何ができる」「いつまでに」「いくら」「何は作らない」が中心
   - 経営者・決裁者・営業担当に渡す用

B. 実務担当者向け（標準・推奨）
   - 機能リスト・用語集・主要な制約まで
   - データ項目名は日本語で表示
   - 技術的背景は最小限（使用する外部サービス名程度）
   - 現場責任者・プロジェクトマネージャーに渡す用

C. エンジニア・情シス部門向け（詳細・技術者向け仕様書に近い）
   - 機能 + 用語集 + データ項目 + API方針 + 認証方式 + 性能要件
   - 技術用語OK、ただし表・図中心で読みやすく整形
   - 相手方エンジニア・情シス部門に渡す用

D. カスタム（粒度を自由記述）
```

**Step 3**: 選択された粒度に応じて、クライアント向けマークダウンを生成

**Step 4**: `docs/NNN_[機能名]_requirements_for_client.md` に出力
- 例: `docs/001_requirements.md` → `docs/001_requirements_for_client.md`
- 番号は技術者向けファイルと揃える（同じものを別フォーマットで出す関係のため）

#### 粒度A（営業・意思決定者向け）のテンプレ差分

- 「主な機能」の技術的な補足を削除
- 「前提・制約」の外部サービス名・詳細を削除（または簡略化）
- 「用語の定義」は残す（提案段階でも揺れ防止は重要）
- 全体を A4 1枚に収まる密度に

#### 粒度C（エンジニア向け）のテンプレ差分

- 「前提・制約」にデータ項目・API方針・認証方式・性能要件を追加
- 「納品物」に技術スタック（使用ライブラリ・バージョン）を追加
- 用語集は技術用語も含めて拡張

## ファイル命名規則（講師の推奨運用準拠）

プロジェクトの `docs/` 配下は番号付きで管理する:

```
docs/
├── 000_PROJECT_STATUS.md       # 進捗管理
├── 001_requirements.md          # 技術者向け要件定義（本スキルの主出力）
├── 001_requirements_for_client.md   # クライアント向け要件定義（フェーズ4で生成）
├── 002_client_request.md        # 打ち合わせ内容・クライアントからの要望
├── 003_database_schema.md       # DB設計
├── 004_meeting_transcript.md    # 打ち合わせ議事録
├── 005_meeting_2_transcript.md  # 2回目以降の議事録
├── 006_task_backlog.md          # タスクバックログ
└── ...
```

機能追加の要件定義は既存番号の次を使う:
- 既存が 006 まで → `007_[機能名]_requirements.md` + `007_[機能名]_requirements_for_client.md`
