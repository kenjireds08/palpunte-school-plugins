# ドキュメント管理ルール

プロジェクトのドキュメントを適切に管理し、セッション間の引き継ぎを円滑にするためのルール。

## 情報の階層構造

| 場所 | 変更頻度 | 内容 |
|------|----------|------|
| CLAUDE.md | 低（ほぼ変わらない） | 技術スタック、プロジェクト概要、ポインター |
| docs/000_PROJECT_STATUS.md | 高（毎日変わる） | シンプルな進捗管理（チェックリスト） |
| docs/001_requirements.md | 低〜中 | 要件定義（/interviewで生成した仕様書） |
| docs/task-backlog.md | 中 | タスク詳細（Phase分割） |
| docs/[機能名].md | 中 | 大きな機能の独立仕様書 |

## CLAUDE.mdの原則

### 書くべきもの
- プロジェクト概要（1-2行）
- 技術スタック（1行）
- 重要な制約やルール
- 詳細ドキュメントへのリンク（ポインター）

### 書かないもの（docs/に切り出す）
- テーブル構成 → `docs/database.md`
- API仕様 → `docs/api.md`
- タスク詳細 → `docs/task-backlog.md`
- 進捗管理 → `docs/000_PROJECT_STATUS.md`

## push時の自動更新ルール

git push 実行時に `000_PROJECT_STATUS.md` を更新する:

1. 作業中タスクの進捗を反映
2. 完了したタスクにチェックを入れ日付を追記
3. 次のタスクを明記

## 膨張防止ルール（200行超えたら）

STATUS.mdが200行を超えたら、以下の手順で圧縮する:

### 1. 完了タスクを圧縮
フェーズ単位で1行にまとめる。

**Before:**
```
- [x] ログイン画面作成（04-01）
- [x] サインアップ画面作成（04-02）
- [x] パスワードリセット画面作成（04-03）
```

**After:**
```
- [x] Phase 1: 認証機能（04-01〜04-03）
```

### 2. 未完了タスクをメインに残す

### 3. 詳細が長いタスクは docs/ に切り出してリンク

## 双方向リンクのルール

ドキュメント間の関連を明確にする:

- **STATUS → 詳細ドキュメント**: `→ 詳細: docs/xxx.md`
- **詳細ドキュメント → STATUS**: 冒頭に `> 対応タスク: 000_PROJECT_STATUS.md の XX`

## プロジェクト作成時の必須ファイル

1. **CLAUDE.md**（最優先）
2. **docs/000_PROJECT_STATUS.md** → テンプレート: `~/.claude/docs/project-status-template.md`
3. **`.claudeignore`**（`.env*`, `*.pem`, `*.key`, `credentials/` を記載）

## セッション開始時チェックリスト

1. CLAUDE.md確認
2. 000_PROJECT_STATUS.md確認
3. git log確認

## セッション終了時のフロー

`/clear-prep` で引き継ぎメッセージを生成 → クリア → 貼り付けて再開。
STATUS.mdが常に最新なら、引き継ぎも円滑に進む。
