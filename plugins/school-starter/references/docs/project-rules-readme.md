# 案件特化ルール置き場

このディレクトリ（`.claude/rules/`）は、**この案件だけに適用したいルール**を置く場所です。
グローバル（`~/.claude/rules/`）のルールとは分離されており、プロジェクト固有のビジネスロジック・コーディング規約・セキュリティ要件などを記述します。

## いつ使うか

- 案件固有のビジネスロジック検証ルール（例: 予約の重複チェック、キャンセル期限）
- クライアントが指定する特殊なコーディング規約
- 案件固有のセキュリティ要件（特殊な暗号化要件、コンプライアンス要件等）

グローバルルール（`~/.claude/rules/` の env-security / development / test 等）は全案件共通。ここに置くのは**この案件だけで有効にしたい**ルールです。

## 書き方（`paths:` フロントマター推奨）

**特定ファイルの編集時だけ**ルールを読み込ませたい場合、`paths:` フロントマターで絞り込めます。Context 消費を節約できます:

```yaml
---
paths:
  - "src/features/booking/**/*.ts"
---

# 予約機能のビジネスロジック
- 予約時刻の重複チェック必須
- キャンセル期限は予約日時の24時間前まで
- 予約上限は capacity_settings テーブルから取得
```

`paths:` を**省略**すれば、すべてのファイル編集時に読み込まれます（グローバル適用）。

## ファイル名の目安

- `booking-rules.md` — 予約のビジネスロジック
- `client-style.md` — クライアント指定のスタイル規約
- `compliance.md` — 業界規制・コンプライアンス要件
- `api-conventions.md` — API 設計の案件独自ルール

## 最初は空で OK

案件を進めて「このルールを毎回 Claude に言うのは面倒」「プロジェクトメンバー全員に守らせたい」と感じたら、このディレクトリに `.md` ファイルを作ってルールを書き出してください。

## 優先度

複数の CLAUDE.md / ルールファイルが存在する場合の優先度:

**`.claude/rules/` > `~/.claude/rules/` > プラグイン配布のルール**

つまり、ここに置いたルールは**他のルールより優先されます**。「全案件ではこうだけど、この案件だけは別ルール」というカスタマイズに使えます。
